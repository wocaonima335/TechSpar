## Q1. Qt 里的父子对象机制为什么能帮助管理内存？

**参考答案：**
`QObject` 支持对象树，子对象在构造时指定父对象后，会被挂到父对象的 children 列表里。父对象析构时会递归析构所有子对象，因此很多堆上创建的 Qt 对象可以把生命周期交给父对象统一管理。它本质上是所有权关系的约定，不是 GC，所以仍然要关注对象是否真的继承 `QObject`、是否适合设置 parent、是否存在跨线程销毁问题。

**追问链路：**
- 如果一个对象不是 `QObject`，你怎么管理它的生命周期？
- 栈上创建的 `QObject` 能不能设置 parent？为什么？
- 跨线程对象销毁时最容易踩什么坑？

**场景题：**
你在一个播放器里创建了很多临时弹窗和控制器对象，程序运行一段时间后内存持续增长。你会先检查对象树、parent 归属，还是先怀疑泄漏工具误报？

## Q2. 信号槽和普通回调相比有什么优势？

**参考答案：**
信号槽最大的价值是解耦。发送者不需要知道接收者的具体实现，只负责发出某个语义事件；Qt 会根据连接关系调用对应槽函数。相比普通函数指针或裸回调，信号槽在类型安全、对象生命周期配合、跨线程队列投递、一个信号连接多个接收者等方面更适合大型 GUI 工程，但也会引入一定运行时开销。

**追问链路：**
- 直接连接和队列连接的区别是什么？
- 跨线程信号为什么默认更安全？
- 信号槽的性能瓶颈一般出现在什么规模？

**场景题：**
一个后台下载线程完成后要同时更新进度条、日志面板和通知弹窗。你会怎么设计信号槽连接，避免线程越界和强耦合？

## Q3. `moc` 是干什么的？为什么 Qt 需要它？

**参考答案：**
`moc` 是 Meta-Object Compiler，用来为带 `Q_OBJECT` 的类生成元对象代码。Qt 很多能力，比如信号槽、运行时类型信息、动态属性、反射式调用，都依赖这套元对象系统。标准 C++ 本身没有完整反射机制，所以 Qt 通过 `moc` 在编译阶段补上框架需要的那部分元信息。

**追问链路：**
- 如果忘了加 `Q_OBJECT`，会出现什么现象？
- `moc` 生成的能力和 C++ RTTI 是一回事吗？
- 为什么 Qt Quick 也离不开这套机制？

**场景题：**
团队里有人说“信号槽不生效就是 Qt bug”。你怎么快速判断是 `moc`、宏、还是编译系统集成问题？

## Q4. 为什么 Qt 不允许在工作线程里直接操作界面？

**参考答案：**
因为大多数 GUI 对象都归属主线程，界面系统本身也依赖主线程事件循环来处理输入、重绘和窗口消息。工作线程直接更新 UI，会破坏线程归属约束，轻则偶发异常，重则崩溃。正确做法通常是让后台线程处理耗时任务，再通过信号槽、`invokeMethod` 或自定义事件把结果切回主线程更新界面。

**追问链路：**
- 如果只是改一个文本标签，为什么也不建议直接跨线程改？
- `invokeMethod` 和信号槽在这里各自适合什么场景？
- 队列连接里的事件是怎么回到主线程的？

**场景题：**
你做了一个文件批量导入功能，后台线程解析完成后直接调用 `setText()` 更新界面，偶发崩溃。你会如何改造？

## Q5. `QThread` 的常见正确用法是什么？

**参考答案：**
工程上更推荐“工作对象 + 线程”的模式：把耗时逻辑写在 `QObject` 派生类里，再用 `moveToThread` 把它迁移到后台线程，通过信号槽启动任务和回传结果。这样线程控制与业务逻辑职责更清晰，也更容易管理生命周期。直接继承 `QThread` 把业务代码塞进 `run()` 不是绝对错误，但在复杂项目里更容易让职责混乱。

**追问链路：**
- `QThread` 对象本身和线程执行体是什么关系？
- `moveToThread` 之后对象的槽函数在哪个线程执行？
- worker 结束后怎么安全释放对象？

**场景题：**
一个视频转码界面需要支持开始、暂停、取消。你会用单个 `QThread` 轮询，还是用 worker + 信号槽？为什么？

## Q6. Qt 的事件循环如果被阻塞，会出现什么问题？

**参考答案：**
主线程事件循环被阻塞后，界面无法响应点击、键盘、重绘、定时器和异步回调，用户感知就是窗口卡死或“未响应”。很多人把问题归结为“Qt 性能差”，其实根因往往是把耗时计算、同步 IO、死循环或大批量数据处理放在了 UI 线程。排查时要从事件循环是否被长时间占用入手。

**追问链路：**
- 如何区分“UI 卡死”和“业务卡死”？
- 定时器超时不触发，第一排查点是什么？
- 事件循环和消息泵在概念上有什么对应关系？

**场景题：**
用户反馈窗口“打开后 3 秒内不能拖动”。如果你只能改一处代码，你优先把哪段同步逻辑移出主线程？

## Q7. Model/View 架构里，为什么数据改了界面却没刷新？

**参考答案：**
常见原因不是 view 有 bug，而是 model 没有按约定发出变更通知。例如自定义 `QAbstractItemModel` 时，如果数据修改后没有正确发 `dataChanged`，增删行时没有配套 `beginInsertRows/endInsertRows` 之类的通知，view 就不知道该刷新哪部分。Model/View 的关键是“数据和通知一起维护”，不是只改底层容器。

**追问链路：**
- `dataChanged` 和 `modelReset` 的代价有什么差别？
- 为什么不建议频繁 `beginResetModel`？
- delegate、model、view 三者的职责边界是什么？

**场景题：**
一个表格里实时更新上千行数据，直接刷新会卡。你会用局部通知、批量更新，还是重建 model？

## Q8. QML 和 Widgets 的核心区别是什么？

**参考答案：**
Widgets 更偏传统桌面控件体系，适合表单、管理后台、工具型客户端；QML/Qt Quick 是声明式 UI，强调属性绑定、状态驱动、动画和组件化，更适合现代交互界面。两者不是简单的新旧替代关系，而是不同 UI 架构思路。实际选型要看项目交互复杂度、团队经验、性能约束以及是否需要与现有 C++ 业务层深度协作。

**追问链路：**
- QML 属性绑定为什么有时会“失效”或循环更新？
- C++ 暴露给 QML 时最常见的接口设计错误是什么？
- Widgets 和 QML 混用时最难的是哪一层的边界？

**场景题：**
一个客户端既要保留老的 Widgets 业务窗口，又要新增动画较多的首页。你会怎么做混合架构设计？

## Q9. Qt 的对象归属线程和销毁顺序为什么很重要？

**参考答案：**
`QObject` 不只是“有父对象就能自动释放”，它还带有线程归属信息。对象在哪个线程创建、事件投递到哪个线程、在哪个线程销毁，都会影响安全性。错误的销毁顺序会导致 queued event 访问已释放对象，或者在线程退出后对象还没清理完。

**追问链路：**
- 为什么 `deleteLater()` 比直接 `delete` 更安全？
- 线程退出前应该先停哪些东西？
- queued connection 里如果接收者已经销毁会怎样？

**场景题：**
你把一个网络请求对象放到了 worker 线程，偶发出现“线程结束后程序才崩”。你会先查对象归属还是查连接释放？

## Q10. Qt 的布局系统为什么比手写坐标更可靠？

**参考答案：**
布局器把不同 DPI、字体、语言、窗口尺寸变化带来的适配问题统一交给框架处理。手写坐标在固定分辨率下看起来没问题，但一旦字体变大、国际化文本变长、窗口缩放或移动到高 DPI 屏幕，就很容易乱掉。布局系统本质上是在做约束求解和空间分配。

**追问链路：**
- 为什么同一套 UI 在中文和英文下会表现不同？
- sizePolicy 和 stretch 的区别是什么？
- 布局嵌套太深会有什么副作用？

**场景题：**
一个表单页面在 2K 屏幕正常，但在笔记本上按钮被挤掉了。你会先调布局、字体，还是改控件尺寸？

## Q11. Qt 的性能问题一般出在哪些地方？

**参考答案：**
高频问题通常不是 Qt “慢”，而是主线程阻塞、频繁重绘、大量信号槽连接、模型刷新粒度过粗、QML 绑定过多等。真正优化要看哪里在占用 UI 线程、哪里触发了过量通知、哪里产生了不必要的对象创建。先定位瓶颈，再决定是拆线程、改刷新粒度还是减少绑定。

**追问链路：**
- 你会先看 CPU、GPU 还是事件循环？
- 为什么“少量对象大量刷新”比“大量对象少量刷新”更危险？
- QML 性能和 Widgets 性能各自最常见的坑是什么？

**场景题：**
一个列表页滚动时明显掉帧，你会优先怀疑图片加载、delegate 创建，还是模型通知风暴？

## Q12. Qt 项目打包发布时最容易漏什么？

**参考答案：**
除了可执行文件本身，还要关注平台插件、动态库依赖、QML 模块、字体、图片资源、数据库驱动和不同系统上的路径差异。很多“本地能跑、拷到别的机器不能跑”的问题，不是代码逻辑，而是打包遗漏或插件搜索路径错误。发布前一定要按干净环境验证。

**追问链路：**
- `platforms` 插件缺失时通常会报什么现象？
- QML 资源和本地文件路径如何区分？
- 你会怎么设计一个最小可复现的发布验证包？

**场景题：**
客户环境里程序启动直接闪退，但开发机正常。你会先查插件、依赖库，还是查配置文件路径？
