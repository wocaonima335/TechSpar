## Q1. TCP 和 UDP 的核心区别是什么？

**参考答案：**
TCP 面向连接，提供可靠传输、按序到达、重传、流量控制和拥塞控制；UDP 无连接，只负责尽力把数据报发出去，不保证一定到、按序到或不重复。TCP 更适合业务正确性优先的系统，UDP 更适合实时性敏感、可接受部分丢失或者由上层自己补齐可靠性的场景。

## Q2. 为什么 TCP 需要三次握手，不能两次？

**参考答案：**
三次握手的目标是让双方都确认“自己能发、也能收”，并同步初始序列号。两次握手只能保证客户端知道服务端收到了请求，但服务端无法确认客户端是否收到了自己的应答。第三次握手补上这一步，同时也能降低历史失效连接请求被误判为新连接的风险。

## Q3. 为什么 TCP 挥手通常是四次？

**参考答案：**
因为 TCP 是全双工连接，两个方向的关闭彼此独立。一方发送 FIN 表示“我这边没数据发了”，对方先回 ACK 确认收到；等对方自己的数据也发完后，再发送 FIN，最后由最初关闭方回 ACK，所以通常表现为四次挥手。它本质上反映的是双向关闭，而不是机械地比握手多一次。

## Q4. `TIME_WAIT` 为什么存在？过多会有什么影响？

**参考答案：**
`TIME_WAIT` 的主要目的是确保最后的 ACK 有机会被对端收到，并让网络里残留的旧报文自然过期，避免污染后续新连接。数量过多会占用本地端口和连接表资源，短连接场景下尤其明显。优化时不能简单粗暴关闭，而要结合连接复用、协议设计和系统参数调整综合处理。

## Q5. 什么是滑动窗口？

**参考答案：**
滑动窗口表示发送方在收到确认前，最多可以连续发送多少数据。它让 TCP 不必一发一等，从而显著提升链路利用率。窗口机制和确认、重传、流量控制紧密相关，是理解高吞吐传输、慢连接和网络抖动问题的基础。

## Q6. 流量控制和拥塞控制有什么区别？

**参考答案：**
流量控制是点对点的，解决“接收方处理不过来怎么办”，核心依据是接收窗口；拥塞控制面向整个网络，解决“链路和中间设备承受不住怎么办”，核心依据是丢包、时延和拥塞窗口变化。一个是保护接收方，一个是保护网络，本质目标不同，但在传输过程中会共同影响发送速率。

## Q7. HTTP 和 HTTPS 的区别只是在于加密吗？

**参考答案：**
不只是。HTTPS 是在 HTTP 之下加了一层 TLS，除了传输加密，还包括服务端身份认证、证书校验、密钥协商以及防止中间人攻击等能力。真正的差异是“通信信任链”被补齐了，而不是单纯把明文换成密文。

## Q8. DNS 查询大致是怎么走的？

**参考答案：**
通常客户端先查本地缓存和系统缓存，没有命中再去问本地域名服务器；本地 DNS 再根据情况做递归查询或迭代查询，逐步找到根域、顶级域、权威服务器，最终拿到记录结果并返回。真实线上访问里，DNS 延迟会直接影响首包时间，所以很多请求慢的问题要先排查解析链路，而不是一上来怀疑业务代码。
