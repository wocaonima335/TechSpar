# ffmpeg 开发

> 第 1 版知识库入口 | 面向音视频工程、播放器、转码和媒体处理方向的 API 编程训练

## 主题定位

ffmpeg 开发训练聚焦 API 使用和媒体处理流程，不把重点放在命令行参数记忆。目标是让训练系统能围绕“封装格式、编解码、时间戳、像素格式、滤镜链、硬件加速”等关键主题出题。

## 1. ffmpeg 组件分工

- `libavformat` 负责封装格式读写、复用和解复用，管理输入输出容器和流信息。
- `libavcodec` 负责编码器、解码器及相关上下文，是音视频数据压缩和解压的核心。
- `libavutil` 提供时间基、像素格式、错误码、字典、缓冲区等通用基础设施。
- `libswscale` 与 `libswresample` 分别处理图像缩放/像素格式转换和音频重采样，是媒体处理常用配套模块。

## 2. 封装与解封装流程

- 封装格式描述的是数据如何组织在文件或网络流里，例如 MP4、FLV、MKV，不等同于编码格式。
- 解封装的核心流程是打开输入、读取流信息、循环读包、按流类型分发 packet。
- `AVFormatContext`、`AVStream`、`AVPacket` 是理解封装层的关键对象。
- 面试经常追问为什么“同样是 H.264，既可以放在 MP4 里，也可以放在 TS 里”，本质就是编码与容器分层。

## 3. 解码流程

- 现代 ffmpeg API 通常采用 `send_packet` / `receive_frame` 模式，而不是旧式一次调用完成所有工作。
- 解码不是“一包对应一帧”，B 帧重排、缓冲和延迟会导致输入输出节奏不同。
- 视频解码核心对象是 `AVCodecContext` 和 `AVFrame`，需要理解 frame 中的数据平面、行跨度和格式信息。
- flush 逻辑不能忽略，流结束时还可能有缓冲帧没有输出。

## 4. 编码流程

- 编码过程一般是配置编码器参数、送入原始帧、循环取出压缩后的 packet。
- 码率、GOP、B 帧、参考帧、像素格式、时间基都会影响画质、延迟和兼容性。
- 视频编码前常要做像素格式转换，音频编码前常要做采样率、声道布局和样本格式转换。
- 低延迟直播和离线转码的参数侧重点不同，这是场景题的高频考法。

## 5. 时间戳与音视频同步

- `pts`、`dts`、`time_base` 是最容易把人绕晕的三组概念，需要结合复用和解码流程理解。
- 不同流的时间基不同，跨模块传递时通常需要 `av_rescale_q` 做时间戳换算。
- 音视频同步本质是选一个主时钟，然后让其他流围绕它做快慢调整或丢帧补偿。
- 时间戳错乱会直接表现为音画不同步、播放卡顿、拖动异常或封装文件不可播。

## 6. 像素格式、采样格式与转换

- 视频常见像素格式包括 `YUV420P`、`NV12`、`RGB24`，要理解平面布局和带宽差异。
- `sws_scale` 负责分辨率和像素格式转换，是播放器截图、转码预处理的常用步骤。
- 音频样本格式、采样率、声道布局不匹配时需要重采样，否则编码器通常无法直接接收。
- 这部分面试很爱问“为什么显示正常但颜色不对”，通常和像素格式解释错误有关。

## 7. 滤镜链与媒体处理

- ffmpeg filter graph 支持把多个处理节点连接起来，例如缩放、裁剪、叠字、去噪、混音。
- 滤镜链的输入输出也依赖格式协商，常见问题是像素格式或时间基不匹配。
- API 层使用滤镜时，需要理解 graph、buffer source、buffer sink 的数据流向。
- 滤镜让复杂处理流水线模块化，但也增加了调试难度和性能开销。

## 8. 硬件加速与工程问题

- 硬件加速常见后端包括 NVDEC/NVENC、VAAPI、DXVA、VideoToolbox，目标是降低 CPU 压力或提升吞吐。
- 引入硬件设备上下文后，frame 可能驻留在 GPU 内存，拷贝和格式转换策略会变得更重要。
- 硬件加速不是无脑更快，初始化成本、设备兼容性、零拷贝链路是否打通都会影响实际收益。
- 音视频工程排查问题时，除了 API 调用顺序，还要关注封装格式、时间戳、线程模型和资源释放是否正确。
