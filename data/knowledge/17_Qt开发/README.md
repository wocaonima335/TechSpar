# Qt 开发

> 第 1 版知识库入口 | 面向 Qt Widgets / Qt Quick / 桌面客户端 / 工业软件方向的专项训练

## 主题定位

Qt 开发训练聚焦“C++ 对象模型 + Qt 框架机制 + GUI 工程实践”的结合，不只考 API 名称记忆。目标是让专项训练能围绕对象树、信号槽、事件循环、线程模型、模型视图、Qt Quick、性能与发布等真实工程问题出题。

## 1. Qt 框架结构与核心模块

- Qt 不只是 GUI 库，还包含对象系统、容器、网络、数据库、多线程、QML、测试等完整应用开发基础设施。
- `QtCore` 提供事件循环、对象树、字符串、容器、文件、线程等核心能力，是绝大多数模块的基础。
- `QtGui`、`QtWidgets`、`QtQuick` 分别服务于底层图形能力、传统桌面控件界面和声明式界面体系。
- 面试常追问“Qt 和标准 C++ 库是什么关系”，关键要能说明 Qt 是框架层补充，不是语言替代。

## 2. QObject、对象树与内存管理

- `QObject` 是 Qt 对象模型核心，支持父子关系、元对象系统、事件机制、信号槽等能力。
- 父对象析构时会递归析构子对象，因此很多 Qt 对象的生命周期可以交给对象树管理。
- Qt 的对象树能减少手动释放负担，但不代表所有对象都能随意设 parent，尤其要区分栈对象、跨线程对象和非 `QObject` 类型。
- 面试高频题是“Qt 为什么很多场景下不需要手动 delete”，回答要落到父子对象和所有权语义，而不是泛泛说“它会自动回收”。

## 3. 信号槽与元对象系统

- 信号槽本质是 Qt 对对象间通信机制的封装，优势在于解耦发送者和接收者。
- `moc` 生成的元对象代码支撑了信号槽、运行时类型信息、动态属性、反射式调用等能力。
- 连接方式包括直接连接、队列连接、阻塞队列连接等，不同线程下行为差异是重点。
- 面试常问“信号槽和普通函数回调有什么区别”，要能从类型安全、线程投递、解耦和运行时开销几个角度说明。

## 4. 事件循环、事件分发与响应机制

- Qt GUI 程序通常由 `QCoreApplication` / `QApplication` 启动事件循环，负责分发输入、定时器、网络和自定义事件。
- 鼠标键盘消息、重绘请求、定时器超时、异步任务结果最终都要回到事件循环里处理。
- `event()`、事件过滤器、具体事件处理函数（如 `mousePressEvent`）构成了常见的事件响应层次。
- 如果主线程被阻塞，界面卡死、本应异步完成的事件也无法及时处理，这是排查 UI 假死的核心知识点。

## 5. 线程模型与界面线程约束

- Qt 的 GUI 对象通常只能在主线程创建和操作，跨线程直接更新界面是典型错误。
- `QThread`、工作对象迁移（`moveToThread`）和信号槽队列连接是常见的后台任务组织方式。
- 线程安全问题不仅是数据竞争，还包括对象归属线程、事件投递、生命周期销毁时机。
- 面试常追问“为什么不建议继承 `QThread` 直接写业务逻辑”，核心在于把线程控制和工作对象职责分离更清晰。

## 6. Widgets、布局系统与绘制

- Qt Widgets 适合传统桌面应用，强调控件树、布局管理、事件响应和自绘能力。
- 布局器解决的是不同分辨率、字体和窗口尺寸下的界面组织问题，不能只靠写死坐标。
- 自定义控件常涉及 `paintEvent`、双缓冲、重绘区域控制和输入事件配合。
- 这部分常考“为什么界面闪烁”“为什么 resize 后布局错乱”“为什么频繁 repaint 导致性能差”。

## 7. Model/View、数据驱动 UI 与代理

- Qt 的 Model/View 体系把数据、展示和交互解耦，典型类包括 `QAbstractItemModel`、`QTableView`、`QListView`。
- 自定义 model 时要特别注意索引、角色、数据变更通知和增删行信号，否则界面不会正确刷新。
- delegate 决定单元格绘制和编辑行为，是复杂表格、树视图的重要扩展点。
- 面试会从“为什么列表数据更新了界面没刷新”切入，考查对通知机制和 model 约束的理解。

## 8. Qt Quick / QML 与 C++ 协作

- QML 提供声明式 UI 描述方式，适合动画丰富、状态驱动明显的现代界面。
- Qt Quick 的关键不是“语法像 JS”，而是场景图、属性绑定、状态机和组件化思想。
- C++ 与 QML 协作时，需要清楚暴露对象、属性、模型和方法的边界，避免让 UI 层承担过多业务逻辑。
- 高频问题包括属性绑定失效、上下文对象滥用、模型角色未声明、跨层调用耦合严重等。

## 9. 网络、数据库与工程集成

- 桌面客户端常结合 `QNetworkAccessManager`、`QTcpSocket`、`QSqlDatabase` 等模块完成业务集成。
- 异步网络请求的关键在于事件驱动回调、超时处理、错误恢复和线程边界，而不是简单发请求收响应。
- Qt SQL 封装提升了跨数据库开发效率，但连接生命周期、线程使用限制和事务控制仍需要明确。
- 工程题往往会结合“播放器客户端”“工业上位机”“IM 工具”等场景考察模块协作能力。

## 10. 性能优化、发布与排障

- Qt 性能问题常出现在主线程阻塞、频繁重绘、大量信号槽连接、模型刷新粒度过粗、QML 绑定过多等场景。
- 发布阶段要关注插件、平台依赖、动态库打包、字体/图片资源、平台差异和部署工具链。
- 排障时要区分逻辑 bug、线程 bug、对象生命周期问题和平台适配问题，不能只盯着崩溃堆栈。
- 一个合格的 Qt 工程回答，通常需要同时给出“机制解释 + 工程后果 + 排查手段”。
