# STL 与容器

> 第 1 版知识库入口 | 面向 C++ 工程与算法实践的标准库使用训练

## 主题定位

STL 与容器关注标准库层面的建模与使用策略，不重复讲 C++ 语言本体。训练重点是“为什么选这个容器”“复杂度代价是什么”“迭代器和算法如何组合”。

## 1. STL 总览与设计思想

- STL 的核心不是几个容器名字，而是“容器 + 迭代器 + 算法 + 仿函数”的组合式设计。
- 标准库追求泛型和零成本抽象，很多性能特征来自类型选择和算法搭配，而不是魔法优化。
- 面试经常会问 STL 为什么把算法和容器解耦，本质是在问抽象层次设计能力。
- 需要熟悉 `begin/end`、范围 for、值语义和异常安全在 STL 中的体现。

## 2. 顺序容器

- `vector` 连续存储、随机访问快，是默认首选容器，但扩容会导致迭代器和引用失效。
- `deque` 支持头尾高效插入删除，适合双端队列场景，但内存布局不连续。
- `list` 和 `forward_list` 插入删除稳定，但缓存友好性差，现代工程里使用频率远低于 `vector`。
- `string` 可以视为字符顺序容器，面试里常与小字符串优化、拷贝成本一起考。

## 3. 关联容器与哈希容器

- `map`、`set` 通常基于红黑树实现，特点是有序、查找复杂度稳定、支持范围查询。
- `unordered_map`、`unordered_set` 基于哈希表，平均查找更快，但最坏情况和哈希冲突要考虑。
- 当题目要求顺序输出、上下界查找、区间遍历时，树形关联容器通常更合适。
- 哈希容器的性能与哈希函数、装载因子、rehash 时机密切相关，不能只背“平均 O(1)”。

## 4. 迭代器模型

- 迭代器是算法和容器之间的桥梁，不同迭代器能力决定了算法能否使用。
- 输入、输出、前向、双向、随机访问迭代器的能力递增关系是高频考点。
- 容器操作后哪些迭代器会失效要单独记忆，特别是 `vector` 扩容、`erase` 删除和哈希表 rehash。
- `const_iterator`、反向迭代器和失效后的未定义行为是实际写代码时最容易踩坑的部分。

## 5. 常用算法库

- `sort`、`stable_sort`、`find`、`binary_search`、`lower_bound`、`upper_bound` 是面试中最常考的一组算法。
- `transform`、`for_each`、`count_if`、`remove_if` 等算法体现的是“用声明式组合替代手写循环”。
- `erase-remove` 惯用法必须理解，因为 `remove` 并不真正删除元素，只是重排并返回新区间尾部。
- 算法复杂度需要和容器特征一起分析，例如对链表排序与对向量排序的代价不一样。

## 6. 容器适配器与特殊结构

- `stack`、`queue`、`priority_queue` 是容器适配器，不直接暴露完整迭代器接口。
- `priority_queue` 默认是大顶堆，比较器方向是很多人写反的经典细节题。
- 适配器底层通常依赖其他容器，例如 `stack` 默认基于 `deque`。
- 适配器适合限制接口、防止误用，体现的是“只暴露需要的能力”。

## 7. allocator 与内存分配

- `allocator` 是 STL 可定制内存管理的基础，目的是把对象构造销毁与原始内存分配解耦。
- 多数业务开发不会手写 allocator，但面试会用它考察对容器底层行为的理解深度。
- 高性能场景下，自定义内存池或 arena 分配往往通过 allocator 风格接口接入标准容器。
- `vector` 扩容、节点容器分配次数、对象构造成本都和分配器行为有关。

## 8. 选型与工程习惯

- 默认优先考虑 `vector`，只有在明确需要有序映射、稳定节点、双端操作等能力时再切换容器。
- 容器选型要同时看时间复杂度、内存占用、缓存局部性、迭代器稳定性和接口语义。
- 尽量复用标准算法而不是手搓循环，能减少 bug 并让意图更清晰。
- 不要把 STL 记成“背表题”，真正的面试回答应该能给出场景、代价和替代方案。
