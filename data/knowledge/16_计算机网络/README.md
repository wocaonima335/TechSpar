# 计算机网络

> 第 1 版知识库入口 | 面向后端、客户端、系统和基础设施岗位的网络基础面试训练

## 主题定位

计算机网络部分以高频面试题为主，强调概念边界、协议流程和工程现象之间的联系。内容不追求教材式全覆盖，而是优先支持“问原理、问流程、问故障表现、问场景取舍”的训练需求。

## 1. 分层模型与协议视角

- OSI 七层和 TCP/IP 四层模型要会说，但面试更看重你能不能把常见协议放到正确层次。
- 应用层关心 HTTP、DNS、WebSocket 等语义；传输层关心 TCP/UDP；网络层关心 IP 路由；链路层关心局域网传输。
- 分层的意义是隔离复杂度，让每层只关心本层职责，但性能优化时经常要跨层观察问题。
- 抓包分析时，分层思维能帮助快速定位问题发生在握手、传输、重传还是应用协议阶段。

## 2. TCP 与 UDP

- TCP 面向连接、可靠传输、按序到达，适合大多数需要正确性的业务系统。
- UDP 无连接、尽力而为、开销更小，适合音视频实时传输、DNS 查询、游戏状态同步等场景。
- 不能简单把 UDP 理解成“更快的 TCP”，因为可靠性、拥塞控制、重传机制都需要业务方补齐。
- 面试常追问“既然 TCP 可靠，为什么还有 UDP”，核心是场景目标不同。

## 3. 三次握手与四次挥手

- 三次握手的目标是确认双方收发能力和初始序列号，避免历史连接请求造成误判。
- 四次挥手比三次握手多一步，是因为 TCP 连接是全双工，两个方向的关闭通常独立发生。
- `TIME_WAIT` 的意义要能讲清，重点是等待旧报文自然消失并保证被动关闭方收到最终 ACK。
- 面试喜欢把 `SYN flood`、半连接队列、`TIME_WAIT` 过多等工程问题和握手挥手放在一起问。

## 4. 滑动窗口、流量控制与拥塞控制

- 滑动窗口决定一次能发送多少未确认数据，是 TCP 提升吞吐的核心机制。
- 流量控制关注接收方是否来得及处理数据，核心信号是接收窗口大小。
- 拥塞控制关注网络是否承受得住流量，常见机制包括慢启动、拥塞避免、快重传、快恢复。
- 丢包不一定等于网络拥塞，也可能来自链路抖动、无线环境或应用处理不及时。

## 5. HTTP 与 HTTPS

- HTTP 是应用层协议，核心是请求/响应语义、方法、状态码、头部、缓存和连接复用。
- HTTP/1.1、HTTP/2、HTTP/3 的差异常从队头阻塞、多路复用、传输层基础上展开。
- HTTPS = HTTP + TLS，不只是“多了一层加密”，还包含证书校验、密钥协商和身份认证。
- 面试通常会追问 TLS 握手大致过程、对称加密和非对称加密各自负责什么。

## 6. DNS 与名字解析

- DNS 的作用是把域名映射为 IP，但真实解析过程往往涉及本地缓存、递归查询和权威应答。
- 记录类型如 A、AAAA、CNAME、MX、TXT 各有用途，面试里最常见的是 A/CNAME 的区别。
- DNS 延迟会直接影响首包时间，很多“网站慢”问题其实不是业务处理慢，而是解析或连接建立慢。
- CDN、负载均衡、容灾切换都常借助 DNS 做流量调度。

## 7. Socket 编程与服务端连接管理

- Socket 是应用访问传输层能力的编程接口，不同语言框架只是对它的进一步封装。
- 服务端典型流程是 `socket`、`bind`、`listen`、`accept`，客户端典型流程是 `socket`、`connect`。
- 阻塞、非阻塞、IO 多路复用、Reactor/Proactor 是网络编程模型的核心概念。
- 面试通常会结合高并发连接、粘包拆包、心跳保活、半包处理来考查对实际系统的理解。

## 8. 常见现象与排障思路

- 访问慢要区分是 DNS 慢、建连慢、TLS 慢、服务端处理慢还是响应体传输慢。
- 大量重传、窗口缩小、连接重置、超时断开都对应不同层面的故障信号。
- Cookie、Session、Token 虽然属于应用层设计，但经常和 HTTP、跨域、鉴权一起混合考察。
- 回答网络题时，最好先讲协议原理，再落到现象、指标和排障动作，这样更像真实工程师的口径。
